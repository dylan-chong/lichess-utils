Architecture Without Managers

Independent modules that watch their own signals

Revised Module Structure

Each module is self-contained
src/
├── main.ts
├── settings/
│ ├── settingsStore.ts (signals + load/save functions)
│ ├── types.ts
│ └── defaults.ts
├── board/
│ ├── watcher.ts (watches DOM, emits boardChanged signal)
│ ├── reader.ts (pure functions: readPiecePositions, getPlayerColor)
│ ├── renderer3d.ts (init/destroy, watches settings, reacts to board changes)
│ ├── dividers.ts (init/destroy, watches dividersEnabled + customBoardEnabled)
│ └── coordinateUtils.ts (pure functions)
├── flash/
│ ├── overlay.ts (init/destroy, watches flashMode settings, reacts to board changes)
│ └── timing.ts (pure functions for calculating intervals)
├── speech/

│ ├── textGenerators.ts (pure functions)

│ ├── overlay.ts (init/destroy, no settings watching)

Module Responsibilities

board/watcher.ts
Purpose: Detect when lichess board changes
Exports: boardChanged signal, init(), destroy()
Watches: Nothing (just observes DOM)
Reacts to: MutationObserver events
Side effects: Increments boardChanged signal
board/reader.ts
Purpose: Read piece positions from lichess DOM
Exports: readPiecePositions(), getPlayerColor(), pixelsToSquare()
Watches: Nothing (pure functions)
Reacts to: Nothing (called on demand)
Side effects: None
board/renderer3d.ts
Purpose: Render 3D board with Three.js
Exports: init(), destroy()
Watches: customBoardEnabled, obfuscationsEnabled, parallax, hoverMode, pieceStyle, blur, blackSegments, blackSegmentsTiming, boardChanged
Reacts to: Settings changes → update rendering; board changes → update piece positions
Side effects: Creates/removes canvas, renders 3D scene
board/dividers.ts
Purpose: Show/hide quadrant divider lines
Exports: init(), destroy()
Watches: dividersEnabled, customBoardEnabled
Reacts to: Settings change → show/hide/reposition SVG
Side effects: Creates/removes/updates SVG element
flash/overlay.ts
Purpose: Control flash mode overlay timing
Exports: init(), destroy()
Watches: flashModeEnabled, flashDuration, flashInterval, boardChanged
Reacts to: Board change → flash briefly; settings change → update timers
Side effects: Shows/hides overlay, manages timers
speech/synthesizer.ts
Purpose: Speak text using Web Speech API
Exports: init(), destroy(), speak(), stop()
Watches: speakRate
Reacts to: Settings change → update speech rate
Side effects: Controls Web Speech API
speech/commands.ts
Purpose: High-level speech commands
Exports: speakQuadrant(), speakAll(), speakColor(), stopSpeaking()
Watches: Nothing (called on demand)
Reacts to: Nothing (orchestrates reader + synthesizer)
Side effects: Calls synthesizer.speak()
drawing/overlay.ts
Purpose: Draw circles and arrows on board
Exports: init(), destroy(), draw(), clear()
Watches: Nothing (manual commands only)
Reacts to: User commands via keyboard
Side effects: Creates/updates/removes SVG drawings

Initialization Pattern

main.ts - No coordinators needed
Responsibilities:
• Wait for lichess DOM
• Load settings from localStorage
• Initialize all modules (each sets up own effects)
• Mount Preact UI
• Set up keyboard command handler
• Return cleanup function for page navigation

Pattern:
• Import module init functions
• Call each init() - they handle their own reactivity
• Store cleanup functions
• On destroy, call all cleanup functions

No coordination needed - each module is autonomous

Data Flow (Revised)

User clicks button:
SettingButton → settings.blur.value = 1
  ↓ (signal change)
board/renderer3d effect() → update blur
  (no other modules care about blur)

User makes move on lichess:
lichess updates DOM
  ↓
board/watcher MutationObserver → boardChanged.value++
  ↓ (signal change)
  ├→ board/renderer3d effect() → update piece positions
  ├→ flash/overlay effect() → flash briefly
  └→ PiecesList component → re-render

User types "pwk" command:
inputHandler detects command
  ↓
speech/commands.speakQuadrant('wk')
  ├→ board/reader.readPiecePositions()
  ├→ speech/textGenerators.generateQuadrantText()
  └→ speech/synthesizer.speak(text)

Key principles:

  • No coordinators - modules are autonomous
  • Each module sets up own effect() in init()
  • Pure functions don't watch anything (called on demand)
  • Modules only watch signals they care about
  • main.ts just calls init() on everything - no orchestration
