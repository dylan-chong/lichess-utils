Settings management approach

You mentioned making settings changes easy to test with minimal code paths

The current userscript has 14 settings (speak rate, pieces list, dividers, custom board, obfuscations, parallax, hover mode, piece style, blur, black segments, timing, flash mode, flash duration, flash interval).

You want "changing settings reapplies all settings for ease of testing" with a minimal code path. How should settings flow through the architecture?

A

Central Settings Store (Recommended)

Single reactive store, components watch settings, any change triggers full reapply

// Flow:
1. User clicks button → SettingsStore.update()
2. Store saves to localStorage
3. Store emits change event
4. ALL components react to new settings
5. Each component reapplies its part (3D board, dividers, blur, etc)

Pros

  • Single source of truth
  • Settings changes always consistent
  • Easy to test: change setting, everything updates
  • No complex invalidation logic

Cons

  • May reapply more than strictly necessary
  • Components need to be idempotent
B

Granular Updates

Components only update when their specific settings change

// Flow:
1. User clicks button → SettingsStore.update('blur')
2. Store determines what changed
3. Only BlurComponent updates
4. Other components stay unchanged

Pros

  • More efficient (no unnecessary work)
  • Better for complex/expensive updates

Cons

  • Complex dependency tracking
  • Harder to reason about state
  • More code paths to test
  • Settings interactions harder to verify

My recommendation: Option A (Central Store) matches your goal of "minimal code path" and "reapply all settings." It's simpler to test and reason about.
