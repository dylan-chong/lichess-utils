Testing with Vitest

Comprehensive testing strategy using the Preact ecosystem standard

Why Vitest for This Project

Preact + Vite ecosystem fit:
• Already using Vite for bundling
• Vitest shares Vite config (no duplication)
• Fast - native ESM, instant re-runs
• Jest-compatible API (familiar)

Built-in features we need:
• happy-dom included (DOM environment for tests)
• TypeScript support out of the box
• Mock utilities (vi.mock, vi.fn)
• Coverage reporting (c8/istanbul)

vs node:test:
• node:test: minimal, no DOM, manual mocking
• Vitest: batteries included, DOM ready, rich ecosystem

Testing Strategy by Directory

pure/ - Pure Functions (Easiest)
What to test:
• Input → output correctness
• Edge cases (empty arrays, boundary values)
• Error handling

No mocking needed

Example tests:
• coordinates.ts: pixelsToSquare() with various positions, white/black perspective
• speechText.ts: generateQuadrantText() with different piece configurations
• commandParser.ts: parseDrawCommand() with valid/invalid inputs
• pieceGrouping.ts: filterQuadrant(), groupByColorAndType()
settings/ - Settings Store
What to test:
• loadSettings() restores from localStorage
• saveSettings() persists correctly
• Signals update properly
• Auto-save effect triggers

Mocking:
• Mock localStorage (vi.stubGlobal)

Example tests:
• Update signal → verify localStorage.setItem called
• Load with stored data → verify signals have correct values
• Load with no data → verify defaults used
dom/ - DOM Operations
What to test:
• boardReader reads correct positions from mock DOM
• boardObserver triggers signal on DOM changes
• Overlays create/update/remove elements correctly

Environment:
• Vitest with happy-dom provides document, querySelector, etc.
• Create mock lichess DOM structure in tests

Example tests:
• boardReader.ts: Create mock pieces with transforms → verify correct squares returned
• boardObserver.ts: Modify mock board → verify boardChanged signal incremented
• overlays/flash.ts: Call show() → verify overlay element visible
• overlays/dividers.ts: Call render() → verify SVG created with correct lines
browser/ - Browser API Wrappers
What to test:
• Wrapper calls correct Web API
• Parameters passed correctly
• Error handling

Mocking:
• Mock Web APIs (speechSynthesis, etc.)

Example tests:
• speechSynthesizer.speak(text, rate) → verify speechSynthesis.speak called
• speechSynthesizer.stop() → verify speechSynthesis.cancel called
• Verify SpeechSynthesisUtterance created with correct text/rate
handlers/ - Business Logic
What to test:
• Orchestration logic
• Correct dependencies called
• Error paths

Mocking:
• Mock DOM operations
• Mock browser APIs
• Mock pure function results (optional)

Example tests:
• handleSpeechCommand('wk'):
  - Mock boardReader.readPiecePositions → return test pieces
  - Mock speechSynthesizer.speak
  - Verify speak called with correct generated text
• update3dBoard(settings):
  - Mock renderer3d methods
  - Verify updateParallax, updateBlur, etc. called with correct values
• handleFlash():
  - Mock overlay and timers
  - Verify flash timing logic (show/hide sequence)
effects/ - Signal Watchers
What to test:
• Signal changes trigger handler calls
• Correct values passed to handlers
• Cleanup functions work

Mocking:
• Mock handlers

Example tests:
• on3dBoard.ts:
  - Mock update3dBoard handler
  - Change customBoardEnabled signal
  - Verify handler called with current settings
• onFlash.ts:
  - Mock handleFlash handler
  - Increment boardChanged signal
  - Verify handler called
components/ - Preact Components
What to test:
• Component renders correctly
• User interactions work (clicks, etc.)
• Conditional rendering based on signals

Tools:
• @testing-library/preact
• @preact/signals for signal manipulation

Example tests:
• SettingButton: Click → verify signal updated
• ControlPanel: customBoardEnabled=true → verify custom board buttons visible
• PiecesList: boardChanged signal → verify component shows updated pieces

Test Patterns Overview

Pure Function Test Pattern
describe('pixelsToSquare', () => {
  it('converts pixels to square for white player', () => {
    const result = pixelsToSquare(195, 390, 58.5, true)
    expect(result).toBe('d5')
  })

  it('flips coordinates for black player', () => {
    const result = pixelsToSquare(195, 390, 58.5, false)
    expect(result).toBe('e4')
  })
})
DOM Test Pattern (with happy-dom)
describe('boardReader', () => {
  it('reads piece positions from DOM', () => {
    // Create mock lichess DOM
    document.body.innerHTML = `
      <cg-board>
        <piece class="white pawn" style="transform: translate(0px, 390px)"></piece>
      </cg-board>
    `

    const pieces = readPiecePositions()
    expect(pieces).toHaveLength(1)
    expect(pieces[0]).toMatchObject({
      color: 'white',
      type: 'pawn',
      square: 'a2'
    })
  })
})
Handler Test Pattern (with mocks)
import { vi } from 'vitest'

describe('handleSpeechCommand', () => {
  it('speaks white kingside quadrant', () => {
    // Mock dependencies
    const mockReader = { readPiecePositions: vi.fn() }
    const mockSynthesizer = { speak: vi.fn() }

    mockReader.readPiecePositions.mockReturnValue([
      { color: 'white', type: 'pawn', square: 'e2' }
    ])

    handleSpeechCommand('wk', mockReader, mockSynthesizer)

    expect(mockSynthesizer.speak).toHaveBeenCalledWith(
      expect.stringContaining('e2 white pawn')
    )
  })
})
Component Test Pattern (with testing-library)
import { render, fireEvent } from '@testing-library/preact'

describe('SettingButton', () => {
  it('cycles through options on click', () => {
    const testSignal = signal(0)
    const { getByRole } = render(
      <SettingButton
        label="Blur"
        signal={testSignal}
        options={[0, 1, 2, 3]}
      />
    )

    const button = getByRole('button')
    expect(button).toHaveTextContent('Blur: 0')

    fireEvent.click(button)
    expect(testSignal.value).toBe(1)
    expect(button).toHaveTextContent('Blur: 1')
  })
})

Summary:

  • Vitest is the Preact/Vite standard - fast, DOM-ready, batteries included
  • All directories are testable (pure, dom, browser, handlers, effects, components)
  • happy-dom provides DOM environment without a browser
  • @testing-library/preact for user-centric component tests
  • Tests live next to source files
  • Run with npm test (once) or npm run test:watch (continuous)
