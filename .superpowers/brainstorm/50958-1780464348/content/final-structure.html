Final Architecture: Organized by Purity

Pure functions, DOM operations, effects, and handlers cleanly separated

Layer Responsibilities

pure/ - Pure Functions
What goes here: Functions with no side effects
Inputs: Plain data (numbers, strings, objects)
Outputs: Plain data
No access to: DOM, signals, localStorage, Web APIs
Testing: Easiest - just call function, check output

Examples:
• pixelsToSquare(x, y, squareSize, isPlayingWhite) → 'e4'
• generateQuadrantText(pieces, 'wk') → "white pawns on e2, f2..."
• parseDrawCommand('-e4,a1b2') → [{circle: 'e4'}, {arrow: ['a1','b2']}]
dom/ - DOM Operations
What goes here: Code that reads/writes DOM
Responsibilities: querySelector, createElement, appendChild, etc
No business logic: Just DOM manipulation
Testing: Skip in unit tests (manual test in browser)

Examples:
• boardReader.readPiecePositions() - queries piece elements, reads transforms
• boardObserver.init() - sets up MutationObserver, updates signal
• overlays/flash.createOverlay() - creates div, appends to cg-container
• renderer3d.updateScene() - updates Three.js meshes
browser/ - Browser APIs
What goes here: Wrappers around Web APIs
Examples: Web Speech API, localStorage, fetch, etc
Why separate: Easy to mock in tests
Testing: Mock the wrapper

Example:
• speechSynthesizer.speak(text, rate) - wraps speechSynthesis.speak()
• speechSynthesizer.stop() - wraps speechSynthesis.cancel()
effects/ - Signal Watchers (Thin Glue)
What goes here: effect() calls that wire signals to handlers
Responsibilities: Watch signals, call handlers, that's it
Keep thin: No business logic, just glue
Pattern: Each file exports init() and destroy()

Structure:
• Reads relevant signals
• Calls handler function with current values
• Returns cleanup function

Example:
on3dBoard.ts watches customBoardEnabled, obfuscationsEnabled, parallax, etc
→ calls update3dBoard() handler with current settings
handlers/ - Business Logic
What goes here: The actual "what to do" code
Called by: Effects (reactive) or commands (imperative)
Can use: Pure functions, DOM operations, browser APIs
Testing: Mock DOM/browser dependencies

Examples:
• update3dBoard(settings) - coordinates renderer3d, applies all settings
• handleFlash(duration, interval) - orchestrates flash overlay timing
• handleSpeechCommand('wk') - reads board, generates text, speaks it

Data Flow Examples

User clicks "Blur: 0px" button
1. components/SettingButton → settings.blur.value = 1
2. effects/on3dBoard effect() detects change
3. effects/on3dBoard calls handlers/update3dBoard(settings)
4. handlers/update3dBoard calls dom/renderer3d.updateBlur(1)
5. dom/renderer3d applies blur filter to canvas
User makes move on lichess
1. lichess updates DOM
2. dom/boardObserver MutationObserver fires
3. dom/boardObserver increments boardChanged signal
4. effects/onBoardChange effect() detects change
5. effects/onBoardChange calls handlers/handleBoardUpdate()
6. handlers/handleBoardUpdate:
   → calls dom/boardReader.readPiecePositions()
   → calls dom/renderer3d.updatePieces(pieces)
7. effects/onFlash effect() also detects boardChanged
8. effects/onFlash calls handlers/handleFlash()
9. handlers/handleFlash orchestrates flash timing with dom/overlays/flash
User types "pwk" command
1. commands/keyboardInput detects "pwk"
2. commands/keyboardInput calls handlers/handleSpeechCommand('wk')
3. handlers/handleSpeechCommand:
   → calls dom/boardReader.readPiecePositions()
   → calls pure/pieceGrouping.filterQuadrant(pieces, 'wk')
   → calls pure/speechText.generateQuadrantText(filteredPieces)
   → calls browser/speechSynthesizer.speak(text, currentRate)

Why this structure works:

  • Testability clear at a glance - pure/ is easy, dom/ is skipped, handlers/ need mocks
  • Effects are thin - just wire signals to handlers, no logic
  • Handlers orchestrate - compose pure functions with DOM/browser operations
  • Pure functions reusable - no coupling to framework or DOM
  • Easy to find code - "where's the coordinate math?" → pure/, "where's the flash logic?" → handlers/
