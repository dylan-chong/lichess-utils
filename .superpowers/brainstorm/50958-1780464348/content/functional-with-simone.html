Functional Architecture for Simone

Module-level functions operating on explicit data types

Simone Constraint

Simone mocks module-level functions, not class methods

Instead of:
• Classes with methods
• Factory functions returning objects with methods

We use:
• Data types (interfaces)
• Module-level functions that operate on those types
• Prefer immutable (return new objects) where possible
• Explicit mutation where unavoidable (DOM, Three.js, timers)

Pattern: Data + Functions

Functions with Explicit Mutation (DOM/Three.js)
// dom/renderer3d.ts

// Data type
interface Renderer3DState {
  scene: THREE.Scene
  canvas: HTMLCanvasElement
  camera: THREE.Camera
  renderer: THREE.WebGLRenderer
}

// Create state
export function createRenderer3D(): Renderer3DState {
  const canvas = document.createElement('canvas')
  const scene = new THREE.Scene()
  // ...
  return { scene, canvas, camera, renderer }
}

// Functions that mutate state (unavoidable with Three.js/DOM)
export function updateBlur(state: Renderer3DState, amount: number): void {
  // Mutates state.scene (Three.js requires mutation)
  state.renderer.setPixelRatio(amount)
}

export function updatePieces(state: Renderer3DState, pieces: Piece[]): void {
  // Mutates scene graph
}

export function destroyRenderer3D(state: Renderer3DState): void {
  state.canvas.remove()
  state.renderer.dispose()
}
Functions with Timers (Explicit State)
// flash/overlay.ts

interface FlashState {
  overlay: HTMLDivElement
  durationTimer: number | null
  intervalTimer: number | null
}

export function createFlashOverlay(): FlashState {
  const overlay = document.createElement('div')
  overlay.className = 'userscript-flash-overlay'
  // ...
  return { overlay, durationTimer: null, intervalTimer: null }
}

export function showFlash(state: FlashState, duration: number): void {
  state.overlay.style.display = 'none' // Show board
  if (state.durationTimer) clearTimeout(state.durationTimer)
  state.durationTimer = setTimeout(() => {
    state.overlay.style.display = 'block' // Hide board
  }, duration * 1000)
}

export function startIntervalFlash(state: FlashState, interval: number, onFlash: () => void): void {
  if (state.intervalTimer) clearInterval(state.intervalTimer)
  state.intervalTimer = setInterval(onFlash, interval * 1000)
}

export function destroyFlashOverlay(state: FlashState): void {
  if (state.durationTimer) clearTimeout(state.durationTimer)
  if (state.intervalTimer) clearInterval(state.intervalTimer)
  state.overlay.remove()
}

Question: When to Mutate vs Return New?

Return new (immutable):
• Pure calculations (coordinates, text generation)
• Data transformations (filtering, grouping)
• Settings (signals already handle immutability)

Mutate in place:
• DOM manipulation (appendChild, remove, style changes)
• Three.js scene graph (adding/removing meshes)
• Timers (storing IDs, clearing)
• Browser APIs (speechSynthesis, MutationObserver)

Reason: These APIs require mutation - we can't "return a new DOM" or "return a new Three.js scene"

Revised Module Patterns

pure/ - Fully Immutable
Pattern: Types + pure functions
State: None (or passed as parameters)
Returns: New values

Example modules:
• coordinates.ts - pixel/square conversions
• speechText.ts - generate text from pieces
• commandParser.ts - parse draw commands
• pieceGrouping.ts - filter/group pieces
dom/ - Explicit State + Mutation
Pattern: State type + functions that operate on it
State: Explicit (canvas, scene, overlay elements)
Mutation: Required (DOM/Three.js APIs)

Example modules:
• boardReader.ts - no state, just reads DOM
• boardObserver.ts - MutationObserver state
• renderer3d.ts - Renderer3DState type + functions
• overlays/*.ts - overlay elements + functions
browser/ - API Wrappers
Pattern: Thin wrappers around Web APIs
State: Managed by browser
Mutation: Calling browser APIs

Example:
• speechSynthesizer.ts - wraps speechSynthesis API
handlers/ - Orchestration Functions
Pattern: Functions that compose other functions
State: Passed from caller
Returns: void (side effects via dependencies)

Example:
• handleSpeechCommand(quadrant, renderer3dState, ...)
• update3dBoard(settings, renderer3dState)
effects/ - Signal Watchers
Pattern: Functions that set up effects
Returns: Cleanup function

Example:
• setup3dBoardEffect(renderer3dState): () => void
• setupFlashEffect(flashState): () => void

Main Entry Point (Revised)

main.ts - Coordinates State
// Create all stateful things
const renderer3dState = createRenderer3D()
const flashState = createFlashOverlay()
const dividersState = createDividers()
const drawingsState = createDrawingsOverlay()
const boardObserverState = createBoardObserver()

// Set up effects (pass state to them)
const cleanup3d = setup3dBoardEffect(renderer3dState)
const cleanupFlash = setupFlashEffect(flashState, boardObserverState)
const cleanupDividers = setupDividersEffect(dividersState)

// Set up commands (pass state as dependencies)
setupKeyboardCommands({
  onSpeechCommand: (cmd) => handleSpeechCommand(cmd),
  onDrawCommand: (cmd) => handleDrawCommand(cmd, drawingsState)
})

// Cleanup function
return () => {
  cleanup3d()
  cleanupFlash()
  cleanupDividers()
  destroyRenderer3D(renderer3dState)
  destroyFlashOverlay(flashState)
  // ...
}

Testing with Simone (Revised)

Mocking Module Functions
import { mockModule } from 'simone'

// Mock entire modules (functions, not classes)
const boardReader = mockModule<typeof import('../dom/boardReader')>('../dom/boardReader')
const renderer3d = mockModule<typeof import('../dom/renderer3d')>('../dom/renderer3d')

it('updates 3d board on settings change', () => {
  // Mock functions called by handler
  renderer3d.expects('updateBlur').withArgs(mockState, 3).returns(undefined)
  renderer3d.expects('updateParallax').withArgs(mockState, 45).returns(undefined)

  // Call handler
  update3dBoard({ blur: 3, parallax: 45 }, mockState)

  // Simone verifies expectations consumed
})

Key changes from earlier design:

  • No classes or factory functions - just data types + module functions
  • Explicit state types - Renderer3DState, FlashState, etc.
  • State created in main.ts - passed to effects and handlers
  • Functions are Simone-mockable - all at module level
  • Honest about mutation - we mutate where APIs require it (DOM, Three.js, timers)
  • Immutable where possible - pure functions return new values
